# 球员录入规范

## 目标

后续真实数据录入以球员为主线推进，采用“核心闭环优先”的节奏：每录一个球员，先让球员详情、关联国家、关联俱乐部、个人奖项和核心荣誉形成可查看、可重算、可校验的一条完整链路。

本规范用于约束后续录入顺序、数据归属和验收方式。最终以页面编辑后的数据库内容为准；脚本只用于赛事结果、批量奖项或明确需要可重放的数据。

## 录入原则

- 先录球员，再补关联实体和核心荣誉，不为了一个球员一次性补完整个世界足球历史。
- 球员、国家、俱乐部、奖项基础信息以页面最终编辑结果为准，避免脚本覆盖人工修正。
- 赛事结果仍按赛事归档，不按球员或俱乐部拆文件。比如梅西的欧冠荣誉写进欧冠赛事，阿根廷荣誉写进对应国家队赛事。
- 俱乐部国内荣誉可以使用 patch 式补录，但仍按赛事归档，例如英超荣誉写进英超补录数据。
- 不确定 UID 的俱乐部不自动创建脏数据；除非明确允许，否则先标记待补或对应名次暂空。
- 隐藏俱乐部用于真实存在但暂不进入豪门列表的实体，默认 `exists=true`、`visibleInCatalog=false`。

## 优先级

| 优先级 | 内容 | 要求 |
| --- | --- | --- |
| P0 必填 | 球员基础资料、代表国家、代表球队/主要俱乐部、主要位置、退役/已故状态 | 每个球员必须完成 |
| P1 优先 | 主要个人奖项、世界杯/洲际杯/欧冠/解放者杯/世俱杯等核心荣誉 | 优先补齐，保证详情页有完整主线 |
| P2 后续 | 国内杯赛、小杯赛、完整俱乐部历史荣誉、非核心奖项、低优先级经历细节 | 后续逐步补齐 |

## 标准流程

### 1. 球员基础资料

先录或核对以下字段：

| 类型 | 字段 |
| --- | --- |
| 身份 | UID、中文名、英文名、外链 |
| 生卒 | 生日、卒日、退役、已故 |
| 国家 | 代表国籍、国籍、足联、出生国家、出生城市 |
| 能力 | 代表位置、位置、PA、CA |
| 身体 | 身高、体重、惯用脚、球衣号、肤色、发色、种族 |
| 扩展 | 球员类型、市场价值、数据库来源、担任过职位 |
| 说明 | 数据来源、成就摘要、备注 |

基础资料允许先不追求完美，但 P0 字段不能空到影响列表和详情识别。

球员列表字段与录入口径：

| 列表字段 | 录入口径 |
| --- | --- |
| UID、球员 | 手动录入 UID、中文名、英文名 |
| PA | 手动录入 |
| 荣誉分 | 由球员奖项规则重算写回，不手填 |
| 年代 | 由生日自动派生，不手填 |
| 生卒年龄 | 由生日、卒日、退役/已故状态展示，不手填 |
| 代表位置、位置 | 手动录入，使用位置标签口径 |
| 身高、体重、球衣、肤色、发色、种族、左右脚 | 手动录入或选择基础配置 |
| 足联 | 由代表国籍/国家关联派生，必要时手动选择关联足联 |
| 代表国籍、国籍、出生城市 | 手动关联国家、国籍列表和城市 |
| 代表球队、初始球队、球队经历 | 通过俱乐部关联和 PlayerCareer 维护 |
| 类型、市场价值、数据库、担任过职位、成就、备注 | 手动录入 |
| 生涯、生命、状态 | 由退役、已故等字段展示，不单独手填 |

奖项数和顶级奖项数由奖项记录和 AwardRule 重算得到，不作为基础资料手动维护。

### 2. 关联实体预检

录球员前先检查这些实体是否存在：

| 实体 | 处理方式 |
| --- | --- |
| 国家 | 不存在则先创建；历史国家按现有历史国家继承口径处理 |
| 俱乐部 | 不存在但确认真实存在时可创建为隐藏俱乐部 |
| 出生城市 | 不存在时按页面能力新增或暂填文本，后续再规范 |
| 奖项 | 不存在则先创建 Award，再录 AwardEdition 和 AwardRecipient |
| 赛事 | 不存在则先按赛事排序规范创建赛事，再补结果 |

如果俱乐部 UID 不确定，不默认填 `-`。先记录到待补项，等 UID 明确后再创建或补 standings。

### 3. 球员经历

球员经历分为国家队经历和俱乐部经历：

- 国家队经历使用 `COUNTRY`，关联对应国家。
- 俱乐部经历使用 `CLUB`，关联对应俱乐部。
- 代表俱乐部只允许一个，设置 `isRepresentative=true`。
- 重要经历设置 `showInProfile=true`，低优先级或短期经历可暂不展示。
- 传奇阵容候选可设置 `isLegend=true`，但不要把所有效力经历都标成传奇。
- `sortOrder` 按时间或重要程度排序，保证详情页经历顺序稳定。

初始球队和代表球队都优先关联已有俱乐部；如果俱乐部存在但不进入豪门列表，仍可以作为经历和跳转对象使用。

### 4. 个人奖项

个人奖项按奖项体系维护：

- Award：奖项本体，例如金球奖、世界足球先生、洲际年度最佳球员。
- AwardEdition：奖项届次，例如 2024 金球奖。
- AwardRecipient：获奖球员、排名、名次、备注和外链。
- AwardRule：奖项分规则，缺失时先补规则，再重算奖项分。

奖项分类、分值、顶级奖项数和组合折减口径见 `docs/球员奖项规则方案.md`。

录入后执行球员奖项规则重算，核对 `awardCount`、`topAwardCount`、`honorScore` 是否符合预期。

### 5. 国家队和俱乐部荣誉

只补该球员直接相关的核心荣誉，不要求一次性补齐所有历史。

国家队优先：

- 世界杯
- 欧洲杯、美洲杯、亚洲杯、非洲杯、金杯、大洋洲杯
- 奥运男足、联合会杯等已纳入规则的核心赛事

俱乐部优先：

- 世俱杯、国际足联洲际杯、欧洲/南美洲杯
- 欧冠、南美解放者杯、亚冠等洲际顶级赛事
- 该球员生涯最关键的国内联赛和主要杯赛

赛事结果录入后执行荣誉规则重算，核对国家和俱乐部的荣誉统计与分数。

### 6. 页面核验

每个球员录完后固定检查：

- 球员列表：姓名、代表国籍、代表球队、位置、PA、状态显示正确。
- 球员详情：基础资料、经历、奖项、生涯摘要无明显缺失或重复。
- 国家详情：代表球员、阵容、国家荣誉统计正常。
- 俱乐部详情：代表球员、阵容、俱乐部荣誉统计正常。
- 巨星荣誉：奖项记录能展示，数量和顶级奖项数正确。
- 国家荣誉和豪门荣誉：新增赛事结果能参与统计，隐藏俱乐部有荣誉时也能出现在荣誉统计中。

## 验证命令

涉及代码或 seed 改动时执行：

```bash
pnpm --filter @total-soccer/api type-check
pnpm lint
```

涉及赛事 seed 时额外执行：

```bash
pnpm --filter @total-soccer/api seed:competitions -- --validate-data <seed-key>
pnpm --filter @total-soccer/api seed:competitions -- <seed-key> --validate-only
pnpm --filter @total-soccer/api seed:competitions -- <seed-key>
pnpm --filter @total-soccer/api seed:competitions -- <seed-key>
```

执行正式 seed 两次是为了确认幂等，不重复创建届次或 standings。

## 样板球员建议

正式大批量录入前，先用 3-5 个样板球员跑完整流程：

| 类型 | 建议样本 | 覆盖点 |
| --- | --- | --- |
| 现役多俱乐部 | 梅西、C 罗 | 多俱乐部经历、国家队荣誉、个人奖项 |
| 退役巨星 | 齐达内、罗纳尔多 | 退役状态、国家队和俱乐部核心荣誉 |
| 已故传奇 | 贝利、马拉多纳 | 已故状态、历史数据、国家队核心荣誉 |
| 欧洲传奇 | 克鲁伊夫、贝肯鲍尔 | 欧冠、欧洲杯/世界杯、历史俱乐部 |

样板球员跑通后，再按相同清单批量扩展。

## 注意事项

- 不要因为录一个球员，把所有相关俱乐部的全部国内荣誉都强制补齐。
- 不要把球员相关荣誉拆成球员文件；赛事结果始终按赛事归档。
- 不要让 seed 覆盖页面手动修正过的名称、别名、外链、UID 等基础信息。
- 不确定来源时宁可留待补项，不要为了数量完整写入可疑数据。
- 每次补完赛事结果或奖项后，要记得重算对应规则，否则页面统计可能不会立刻变化。
