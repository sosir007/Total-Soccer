# 天下足球可行性方案与实施计划

## 1. 可行性结论

### 1.1 总体结论

项目可行，适合按“前后端分离 + 模块化单体后端 + PostgreSQL 主数据库”的方式实施。

当前需求的核心难点不是高并发，而是数据建模、表格导入、评分规则抽象、统计口径一致性和后续可维护性。第一阶段不建议直接上微服务，也不建议把所有统计逻辑写死在前端或临时脚本里。更稳妥的路线是先做一个边界清晰的模块化单体，等数据规模、团队规模和访问量上来后再拆服务。

### 1.2 适合先做的 MVP

第一版 MVP 建议聚焦：

- 从 `.lakesheet` 导入数据。
- 建立球员、国家、俱乐部、荣誉规则、基础配置等核心数据模型。
- 完成球员/国家/俱乐部列表与详情。
- 完成天下纵览下的世界概览。
- 完成紫禁之巅基础排行。
- 完成基础筛选、排序、分页。
- 完成荣誉分规则配置和手动重算。
- 完成后台管理入口。

暂缓内容：

- 公开分享页。
- 高级阵容编辑器。
- 数据版本对比。
- 复杂趋势图。
- 多端适配 App。
- 微服务拆分。

## 2. 推荐技术架构

### 2.1 总体架构

```text
Browser
  |
  | HTTPS
  v
Vue3 Web App
  |
  | REST API / JSON
  v
NestJS API Server
  |
  | Prisma ORM
  v
PostgreSQL
  |
  +-- Redis: 缓存、异步任务状态、限流
  +-- Object Storage: 导入文件、错误报告、导出文件
  +-- Worker: 导入解析、统计重算、荣誉分重算
```

### 2.2 推荐技术栈

| 层级 | 推荐方案 | 原因 |
| --- | --- | --- |
| 前端框架 | Vue3 + Vite + TypeScript | 更贴合当前开发者技术栈，启动快，工程简单，生态成熟 |
| 路由 | Vue Router | Vue 官方路由方案，适合后台多页面模块 |
| 状态管理 | Pinia | Vue 官方推荐状态管理方案，类型体验好 |
| UI 组件 | Element Plus / Naive UI 二选一 | Element Plus 适合后台表格密集场景；Naive UI 更适合定制视觉 |
| 图表 | ECharts | 中文生态成熟，适合 PA 分布、位置分布、荣誉排行 |
| 后端框架 | NestJS + TypeScript | 模块化清晰，适合业务复杂的 API 服务 |
| ORM | Prisma | 类型安全、迁移体验好、适合中小团队快速落地 |
| 主数据库 | PostgreSQL | 关系数据、统计查询、JSONB 扩展能力都适合本项目 |
| 缓存 | Redis | 缓存看板统计、异步任务状态、接口限流 |
| 文件存储 | S3 兼容对象存储 / 本地 MinIO | 存放导入文件、错误报告、导出文件 |
| 搜索 | PostgreSQL 全文/模糊查询，后续可接 Meilisearch | 第一阶段无需引入重型搜索服务 |
| 任务队列 | BullMQ + Redis | 处理导入解析、荣誉分重算、统计聚合 |
| 部署 | Docker Compose 起步，后续 Kubernetes | 初期简单可控，后续可扩展 |

### 2.3 为什么不建议第一阶段微服务

- 业务边界还在演化，过早拆分会增加接口、部署、调试和数据一致性成本。
- 当前数据规模更像“资料库 + 统计系统”，不是天然高并发交易系统。
- 导入、规则、统计之间依赖较强，单体内事务和重算更容易保证正确。

推荐做法是“模块化单体”：

- 代码按模块拆：player、country、club、honor、dashboard、import、config、auth。
- 数据库先统一。
- 后续如果导入或计算压力变大，再把 import worker、calculation worker 独立部署。

## 3. 前端方案

### 3.1 前端应用形态

建议做一个后台/资料库混合型 Web 应用，而不是营销站。

页面风格应偏数据工具：

- 信息密度较高。
- 表格、筛选、标签、图表清晰。
- 不做大面积宣传型 Hero。
- 首屏直接进入天下纵览下的世界概览。

### 3.2 页面模块

| 一级目录 | 二级页面 |
| --- | --- |
| 天下纵览 | 世界概览、紫禁之巅 |
| 国家圣殿 | 国家概览、国家荣誉、国家详情 |
| 豪门殿堂 | 豪门概览、豪门荣誉、豪门详情 |
| 巨星殿堂 | 巨星概览、巨星荣誉、巨星详情 |
| 天机阁 | 荣誉规则、数据导入、基础配置、备注管理、权限管理 |

### 3.3 前端状态设计

所有核心页面都要覆盖：

- loading：骨架屏或表格 loading。
- empty：无数据时引导导入。
- error：接口失败，提供重试。
- permission：无权限时提示申请权限。
- disabled：导入中、重算中、保存中禁止重复操作。
- success：保存、导入、重算成功提示。

### 3.4 前端数据请求

推荐：

- Vue Router 管理页面路由，并将筛选、排序、分页状态同步到 URL Query。
- Pinia 管理登录用户、权限、全局配置、当前导入任务等跨页面状态。
- Axios 或 ky 作为 HTTP client，统一处理 token、错误码、requestId。
- VueUse 用于常见组合式能力，例如防抖、窗口尺寸、localStorage、复制等。
- 表单校验使用 Element Plus Form Rules，复杂场景可引入 VeeValidate + Zod。
- 列表页请求参数统一为：`page`、`pageSize`、`keyword`、`filters`、`sort`。

### 3.5 UI 组件建议

如果目标是尽快落地后台：

- 首选 Element Plus。
- 表格、表单、筛选器、弹窗、上传、权限状态等能力完整，中文资料多。

如果目标是做更有风格的公开资料站：

- 首选 Naive UI。
- 视觉定制空间更大，但需要多做一些业务组件封装。

当前阶段推荐：Vue3 + Vite + TypeScript + Element Plus + ECharts + Pinia。

### 4.1 后端模块

| 模块 | 职责 |
| --- | --- |
| AuthModule | 登录、用户、角色、权限 |
| PlayerModule | 球员 CRUD、列表筛选、详情、荣誉关联 |
| CountryModule | 国家 CRUD、国家汇总、国家荣誉 |
| ClubModule | 俱乐部 CRUD、俱乐部汇总、俱乐部荣誉 |

| DashboardModule | 世界概览、PA 分布、位置分布、排行榜 |

### 4.2 API 风格

第一阶段推荐 REST API。

原因：

- 管理后台数据交互直接。
- 缓存和调试简单。
- 前后端团队协作成本低。
- 未来需要复杂聚合时再局部引入 GraphQL 也不迟。

统一返回结构：

```json
{
  "data": {},
  "message": "ok",
  "requestId": "xxx"
}
```

分页返回结构：

```json
{
  "data": {
    "items": [],
    "page": 1,
    "pageSize": 20,
    "total": 100
  },
  "message": "ok",
  "requestId": "xxx"
}
```

### 4.3 异步任务

以下操作必须异步化：

- lakesheet 文件解析。
- 大批量导入。
- 荣誉分重算。
- 看板统计重算。
- 错误报告生成。

任务状态：

- pending
- parsing
- validating
- waiting_confirm
- importing
- recalculating
- success
- failed
- canceled

当前导入 MVP 已按“预览后确认”的方式落地：`ImportTask.previewData` 保存解析后的工作表数据，确认导入时优先写入 `Country`、`Club`、`Player`、`Position`、`PlayerType` 等基础实体；源数据工作表仍使用 `国家汇总`、`俱乐部汇总`、`球员总表`、`base` 等原始名称，前端展示继续映射为国家圣殿、豪门殿堂、巨星殿堂、天机阁等产品命名。

## 6. 数据导入方案

### 6.1 lakesheet 解析

当前已先落地导入预览 MVP：`POST /import-preview` 接收 multipart 文件，解析 lakesheet、Excel 或 CSV，返回工作表概览、样例数据和基础错误报告，并写入 `import_tasks`、`import_errors`。正式入库、事务回滚和统计重算后续再补。

当前 `.lakesheet` 文件是 JSON，内部 `sheet` 字段是 zlib 压缩字符串。解析流程：

1. 读取 JSON。
2. 检查 `format = lakesheet`。
3. 将 `sheet` 字段按字节还原为 Buffer。
4. zlib inflate 解压。
5. 解析出工作表数组。
6. 从 `data[row][col].v` 提取单元格值。
7. 如果单元格值是公式对象，读取 `value`。
8. 如果单元格值是链接对象，保留 `text` 和 `url`。

### 6.2 字段映射

每个工作表建立固定 mapping：

- `球员总表` -> players、player_positions、player_countries。
- `球员汇总` -> player summary stats。
- `国家汇总` -> countries、country_player_stats。
- `国家荣誉` -> country_honors。
- `俱乐部汇总` -> clubs、club_player_stats。
- `俱乐部荣誉` -> club_honors。
- `荣誉` -> honor_rules。
- `base` -> positions、player_types、potential_rules、ability_value_rules。
- `备注` -> remarks。

### 6.3 导入校验

必须校验：

- 必填字段缺失。
- UID 重复。
- UID 类型不合法。
- PA/CA 不是数字。
- 日期不可解析。
- 国家或俱乐部引用不存在。
- 荣誉规则无法匹配。
- 同一数据行重复导入。

### 6.4 数据版本

建议每次导入生成一个 `data_version`：

- 支持查看当前数据来自哪次导入。
- 支持回滚到上一版本。
- 支持后续做版本对比。

第一版可以只做导入日志和软回滚，完整版本对比放到二期。

## 7. 荣誉分计算方案

### 7.1 计算原则

- 规则配置化，不写死在代码里。
- 每次计算生成 calculation run。
- 每个目标对象保留计算明细。
- 规则变更后允许重算。

### 7.2 计算公式

基础公式：

```text
score = baseScore * placementWeight * coefficient * count + bonus
```

其中：

- `baseScore`：赛事基础分。
- `placementWeight`：冠军、亚军、季军、殿军权重。
- `coefficient`：赛事、地区、联赛系数。
- `count`：次数。
- `bonus`：附加分。

### 7.3 计算结果

需要保存：

- 目标对象。
- 荣誉规则。
- 原始次数。
- 使用的基础分。
- 使用的权重。
- 使用的系数。
- 计算得分。
- 计算批次。

这样后续用户可以解释“为什么这个国家/俱乐部/球员是这个分数”。

## 10. 主要风险与应对

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| lakesheet 格式变化 | 导入失败 | parser 独立封装，保留原始文件和解析日志 |
| 荣誉规则复杂 | 分数不可信 | 规则配置化，保存计算明细，支持重算 |
| 表格公式口径不清 | 统计偏差 | 导入时保留公式结果和原始行，关键统计由后端统一计算 |
| UID 不稳定或重复 | 关联错误 | 建立导入校验和人工确认流程 |
| 页面表格过多 | 前端体验差 | 分页、列配置、筛选折叠、默认视图 |
| 后续需求膨胀 | 工期失控 | MVP 锁定资料库、看板、导入、规则四条主线 |


### 11.3 第一优先级

第一优先级不是页面，而是数据闭环：

1. 能解析 lakesheet。
2. 能落库。
3. 能查询。
4. 能统计。
5. 能重算。

只要这条链路打通，前端页面可以逐步丰富；如果数据链路不稳，页面做得再漂亮也难以维护。
